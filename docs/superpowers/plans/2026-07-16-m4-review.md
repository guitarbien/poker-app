### Task 2: review/replay.ts（回放重建）

**Files:** Create `src/review/replay.ts`、`src/review/replay.test.ts`

**Interfaces:**

```ts
// 由 HandRecord 重建可重放的牌局
// 原理：發牌順序固定（button+1 起每人 2 張），board 接續其後——
// 組出 riggedDeck 後 newHand 即可完全重現初始狀態（同測試的 rigged 手法）
export function replayDeck(record: HandRecord): Card[];
// states[i] = 第 i 個動作「執行前」的 GameState；states[actions.length] = 終局
export function replayStates(record: HandRecord): GameState[];
```

實作要點：
- `replayDeck`：依 `record.button` 與 `record.players`（seat 升冪）算發牌順序——`(buttonIdx + n + 1) % len`（n = 0..len-1，從 button **下一位**開始；此公式自動涵蓋 HU），**每位玩家連續取 2 張**（engine 是 `deck.shift()` 兩次，非輪流一張；無 burn card），deck = [各家 hole 依發牌序成對展平, ...record.board, ...其餘 52 張任意序]
- `replayStates`：`newHand({ players: record.players 映射, button, blinds, handNumber, deck: replayDeck(record) })` 為 `states[0]`，逐一 `applyAction(states[i], record.actions[i].action)` 推進
- 驗證不變量：重放結束的 `board` 應等於 `record.board`（run-out 也一致，因 board 牌就在 deck 對應位置）

- [ ] **Step 1: 失敗測試**：用引擎實打一手 rigged 對局（比照既有測試 helper）→ `buildHandRecord` 產 record → `replayStates`：①`states[0]` 各家 hole 與 record 相同 ②每步 `potAfter` 與 record.actions[i].potAfter 相同（用 Σ totalCommitted 驗）③終局 board 與 stacks 等於原終局 ④含一手有 fold 提前結束與一手打到攤牌的案例
- [ ] **Step 2–4: TDD 循環**；coverage include 已含 `src/review/**`

### Task 3: review/grader.ts（決策標記）

**Files:** Create `src/review/grader.ts`、`src/review/grader.test.ts`

**Interfaces:**

```ts
import type { ReviewFlag } from './recorder';

// 對一筆 HandRecord 評分（human = players.find(p => !p.isCpu)）；rng 注入供 equity 用
// 分子與分母同源產出，避免偵測邏輯寫兩份造成 drift
export interface GradeResult {
  flags: ReviewFlag[];
  opportunities: { rfi: number; postflopCall: number };
}
export function gradeHand(record: HandRecord, rng: Rng): GradeResult;

// 彙總（holdem.reviewFlags v1）
export type FlagAggregates = Record<ReviewFlag['kind'], { count: number; opportunities: number }>;
export const EMPTY_AGGREGATES: FlagAggregates;
export const FLAGS_KEY = 'holdem.reviewFlags';
export const FLAGS_VERSION = 1;
// preflop-loose 與 preflop-tight 兩個 key 的 opportunities 都累加同一個 rfi 數；
// call-without-odds 的 opportunities 累加 postflopCall
export function accumulate(prev: FlagAggregates, result: GradeResult): FlagAggregates;
```

評分規則（spec §9b；用 `replayStates` 取每個決策點狀態）：
- **RFI 判定**：human 的 preflop 動作，且該決策點 `state.currentBet === state.blinds.bb`（無人加注）、`positionOf(state, 0) !== 'BB'`：
  - raise 且 `isInRange(RFI_RANGES[pos])` → 標準；raise/call 且不在表 → `preflop-loose`；fold 且在表 → `preflop-tight`；fold 且不在表 → 標準；call（limp）且在表 → 標準（不鼓勵但不標記）
  - 無論標記與否都算一次 RFI opportunity
- **翻牌後跟注**：human 在 flop/turn/river 的 `call` 動作：`estimated = equity(hole, board, 該決策點未棄牌對手數, 2000, rng)`、`required = owe / (pot + owe)`（pot = Σ totalCommitted、owe = currentBet − committed，皆取決策點狀態）；`estimated < required - 0.02` → `call-without-odds`，`detail: { requiredEquity, estimatedEquity }`（四捨五入 3 位）。每次跟注算一次 opportunity
- `actionIndex` = record.actions 的索引

- [ ] **Step 1: 失敗測試**（固定 seed）：①UTG 拿 72o open raise → preflop-loose ②UTG 拿 AA fold → preflop-tight ③UTG 拿 AJo open → 無標記但 rfi opportunity+1 ④BB 情境不評 ⑤面對加注的 preflop 跟注不評（非 RFI）⑥**前有 CPU limp 時 human 的決策不評分、不計 opportunity** ⑦river 賠率不足跟注 → call-without-odds 且 detail 數字合理：**手牌必須用「只能玩 board 的死牌」如 32o on KQJ87**（required = 1/3、estimated ≈ 0.06；注意 A2o 這種 ace-high 實測 equity ≈ 0.34 反而不會被標——選牌陷阱）⑧成牌強勢跟注不標記但 postflopCall opportunity+1 ⑨accumulate 的分子分母（loose/tight 共用 rfi 分母）
- [ ] **Step 2–4: TDD 循環 + coverage**
- [ ] **Step 5**: Commit：`新增決策評分：RFI 對照與賠率不足跟注標記`
